通信系の自社開発案件

2024/7 - 現在

自社サービスの配信システムのAWS技術検証

開発環境：Python, AWS(API Gateway, SQS, Lambda, RDS, ElastiCache, CloudWatch, X-Rayなど)

自社は通信サービスを運用している。インフラ構成をマイクロサービスにすることで、障害発生時のサービスへの影響を最小限にする狙いがあった。モデルを作成し、システムの実現可能性などを検証した。 具体的には、電話番号とメッセージ内容に応じて処理を分けていく。その過程で、外部サービスのAPIを呼び出す必要があるが、呼び出し制限がある。そのために、ElastiCacheを使用して1秒当たりの呼び出し制限を実装した。 主に以下の項目について、様々なパラメータとともに検証した。

  • (1). パラメータに応じて適切なLambda処理に分岐できるか
  • (2). APIの呼び出し制限に合わせて、意図した処理エラーを返せるか
  • (3). 失敗したLambdaについて適切に再実行できるか

通信系の自社開発案件

2022/6 - 2023/9

数億レコードの通話情報の集計Webアプリのリプレイス

開発環境：Laravel9, Vue.js, MySQL, Docker, Github

3名のチーム（プロジェクトリーダー1名、シニアエンジニア1名、若手エンジニア1名）において、 バックエンドエンジニアの1メンバーとしてプロジェクトに入り、主にコーディングとテスト工程を担当してきた。

私がチームに入った時は、設計書があるのみでコードは存在しなかった。そのため、環境構築やコーディングルールも明示されていなかった。 既存のコードがないからこそ、最初にきちんとルールを決めておく必要があると感じ、最初のころはシニアエンジニアの方にレビューをお願いしていた。 また、開発環境の構築にDockerを用いたことで、新しく入ったエンジニアでも、円滑に構築できた。 さらに、phpunitで十分な単体テストを書くことで、コードの品質を安定させた。 精算処理にかかわるバッチ処理では、複数のテーブルを結合し、集計する複雑な処理だったため、 シニアエンジニアでも設計を間違えていることがありました。少しでも気になることは、質問するように心がけていた。 また、設計書におかしな点があった時には、そのことを伝えるだけでなく、どのように修正したらよいかについて、自分の意見を伝えるようにしています。 その結果、設計書に新しい機能を追加する業務なども任せていただけるようになりました。チームを引っ張るようなマネージャー的な役割は果たしていないが、 指示待ちではなく主体的に動くように意識しています。
